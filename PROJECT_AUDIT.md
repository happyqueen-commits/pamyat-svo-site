# Анализ проекта `pamyat-svo-site`

Дата анализа: 2026-03-26.

## Что сделано хорошо

1. **Понятная доменная модель в Prisma**
   - Есть разделение на `User`, `Session`, `Person`, `Submission`, `CorrectionRequest`, `AuditLog` и т.д.
   - Выделены enum-ы для статусов, ролей и типов медиа, что снижает хаос со строками в коде.

2. **Базовая безопасность аутентификации реализована корректно для MVP**
   - Сессионный токен генерируется криптографически (`randomBytes`), хранится в БД в виде SHA-256 хеша, а в cookie кладётся только «сырой» токен.
   - Cookie помечается как `httpOnly` и `sameSite: 'lax'`.
   - Пароли хэшируются через `scrypt` с солью и проверяются через `timingSafeEqual`.

3. **Валидация входных данных на уровне API**
   - Для ключевых форм (`submission`, `auth`, `role update`, `correction status`) используется Zod.
   - Ввод проходит `safeParse`, ошибки возвращаются пользователю с понятными сообщениями.

4. **Есть зачатки операционной зрелости**
   - Введён audit-log для действий входа/выхода/модерации/смены ролей.
   - Есть ограничение частоты запросов (логин, регистрация, отправка материалов).
   - Есть seed-скрипт с тестовыми пользователями и демонстрационными данными для быстрого старта.

5. **Сильная визуальная база и единый стиль интерфейса**
   - Используется единая система дизайн-токенов (цвета, радиусы, тени) в `globals.css`, благодаря чему экраны выглядят целостно.
   - Есть понятная иерархия «hero → карточки → секции», что хорошо работает для контентного мемориального проекта.
   - Тональность интерфейса выдержана: аккуратные «мягкие» карточки, спокойная цветовая гамма, визуальный акцент на смысле контента.

## Что не так / риски

1. **`next build` «успешен», но с runtime-ошибками Prisma при пререндере**
   - Во время сборки серверные страницы обращаются к таблицам, которых может не быть до `db push/seed`; в логах появляются ошибки `The table main.Person does not exist`, `main.Story`, `main.TimelineEntry`.
   - Это означает, что сборка не полностью «чистая» и поведение окружения сильно зависит от состояния локальной SQLite.

2. **Rate limit хранится в памяти процесса**
   - Лимитер реализован через `Map` в `globalThis`, что работает только в рамках одного процесса.
   - В serverless/multi-instance сценариях лимит будет обходиться, а после рестарта — сбрасываться.

3. **Поиск в архиве сделан в памяти, а не в БД**
   - Сначала загружаются все подходящие карточки, потом фильтрация по `q` идёт на Node.js.
   - На росте данных это даст лишнюю нагрузку на память/CPU и ухудшит latency.

4. **Нет централизованной CSRF-защиты для state-changing endpoints**
   - Есть `sameSite=lax`, но нет явного CSRF-токена/проверки origin для POST/PATCH API.
   - Для кабинета и административных маршрутов это желательно добавить.

5. **Загрузка файлов проверяет размер, но почти не ограничивает тип/содержимое**
   - `detectMediaType` ориентируется на `file.type`, который можно подделать на клиенте.
   - Файлы пишутся в `public/uploads/submissions`, то есть сразу становятся веб-доступными.

6. **Нет установленного ESLint-конфига (интерактивный `next lint`)**
   - Команда `npm run lint` запускает мастер настройки вместо линтинга, значит единый автоматический quality gate пока не настроен.

## Визуальная составляющая и user-friendly: что улучшить

Ниже — прицельные UX/UI-наблюдения по текущим экранам (`/`, `/archive`, `/persons/[id]`, форма авторизации и общая шапка/навигация).

1. **Слишком плотная шапка на средних ширинах**
   - В хедере одновременно длинный бренд, 5 пунктов меню и пользовательские действия.
   - На ноутбуках это может визуально «сжимать» интерфейс, ухудшая сканируемость.
   - Что сделать:
     - На брейкпоинте планшет/ноутбук вынести часть пунктов в «Ещё».
     - Сократить подпись бренда в одну строку на узких ширинах.
     - Добавить явное мобильное меню (если его пока нет).

2. **Недостаточная доступность состояний фокуса**
   - Есть `:hover` для ссылок, но визуально явный `:focus-visible` должен быть более контрастным и заметным для клавиатурной навигации.
   - Что сделать:
     - Ввести единый focus-ring (например, контрастный outline/box-shadow) для `a`, `button`, `input`, `select`, `textarea`.
     - Проверить контраст на светлых фонах у вторичных кнопок и мета-текста.

3. **Не хватает «быстрых подсказок» в ключевых сценариях**
   - Формы (`AuthForm`, `CorrectionForm`) дружелюбные, но не всегда объясняют «что будет дальше».
   - Что сделать:
     - Добавить короткий текст под CTA: срок обработки, кто увидит данные, что происходит после отправки.
     - Для ошибок/успеха использовать единый формат alert-блоков (цвет + иконка + краткий текст).

4. **Карточки в архиве могут терять информативность при длинных биографиях**
   - Сейчас обрезка по фиксированной длине, но без выравнивания по высоте/типографике карточки могут «прыгать» по восприятию.
   - Что сделать:
     - Добавить `line-clamp` для отрывка и фиксировать высоту карточки в сетке.
     - Явно выделить 1–2 ключевых атрибута (например, роль и регион) в одинаковой зоне карточки.

5. **Слабая явность текущего раздела в навигации**
   - В верхнем меню нет явного active-состояния текущей страницы.
   - Что сделать:
     - Подсвечивать активный пункт (цвет/фон/подчёркивание) через pathname.
     - Для длинных страниц добавить «якорную» локальную навигацию (например, в карточке персоны: «Биография / Истории / Фото / Документы»).

6. **Формы можно сделать заметно «доброжелательнее»**
   - В текущем виде поля без явных `required`-маркеров и без inline-подсказок валидации до отправки.
   - Что сделать:
     - Добавить `required`, `aria-invalid`, `aria-describedby`.
     - Подсвечивать ошибки под конкретным полем, а не только общим сообщением.
     - В `textarea` показать счётчик символов и пример структуры сообщения.

7. **Состояние «пусто» и «ничего не найдено» можно усилить**
   - Для пустого результата в архиве сейчас короткая фраза — функционально ок, но UX можно сделать теплее.
   - Что сделать:
     - Добавить быстрые действия: «сбросить фильтры», «посмотреть все роли», «предложить материал».
     - Показать примеры поисковых запросов рядом с полем поиска.

## Приоритет UX-улучшений (быстрые победы)

1. **P0 (быстро и сильно влияет):**
   - active-состояние меню;
   - заметный focus-ring;
   - более явные тексты успеха/ошибок в формах.

2. **P1:**
   - адаптация шапки на средних ширинах;
   - улучшение empty-state в архиве;
   - `line-clamp` и визуальное выравнивание карточек.

3. **P2:**
   - якорная навигация внутри страницы персоны;
   - прогрессивные подсказки и микро-копирайт в формах.

## Рекомендуемый приоритет исправлений

1. **P0:** гарантировать «чистую» сборку без Prisma-ошибок (инициализация БД до рендера, graceful fallback на пустые данные, health-check readiness).
2. **P1:** вынести rate limit в Redis/Upstash/DB с TTL.
3. **P1:** перевести поиск в `/archive` на SQL-фильтрацию (`contains`, индексы, пагинация).
4. **P1:** добавить CSRF-strategy (origin check + anti-CSRF token для мутаций).
5. **P2:** усилить upload security (allowlist расширений, проверка magic bytes, антивирусный pipeline, отдельный storage bucket).
6. **P2:** зафиксировать ESLint/TypeScript quality gate в CI.

## Итог

Для MVP проект выглядит структурно аккуратно: хорошая базовая схема, ролевая модель, модерация, аудит и валидация уже есть.

Главные слабые места — эксплуатационные: устойчивость сборки/рантайма при пустой БД, масштабируемость rate-limit/поиска и hardening безопасности для production-нагрузки.
