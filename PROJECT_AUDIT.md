# Анализ проекта `pamyat-svo-site`

Дата анализа: 2026-03-26.

## Что сделано хорошо

1. **Понятная доменная модель в Prisma**
   - Есть разделение на `User`, `Session`, `Person`, `Submission`, `CorrectionRequest`, `AuditLog` и т.д.
   - Выделены enum-ы для статусов, ролей и типов медиа, что снижает хаос со строками в коде.

2. **Базовая безопасность аутентификации реализована корректно для MVP**
   - Сессионный токен генерируется криптографически (`randomBytes`), хранится в БД в виде SHA-256 хеша, а в cookie кладётся только «сырой» токен.
   - Cookie помечается как `httpOnly` и `sameSite: 'lax'`.
   - Пароли хэшируются через `scrypt` с солью и проверяются через `timingSafeEqual`.

3. **Валидация входных данных на уровне API**
   - Для ключевых форм (`submission`, `auth`, `role update`, `correction status`) используется Zod.
   - Ввод проходит `safeParse`, ошибки возвращаются пользователю с понятными сообщениями.

4. **Есть зачатки операционной зрелости**
   - Введён audit-log для действий входа/выхода/модерации/смены ролей.
   - Есть ограничение частоты запросов (логин, регистрация, отправка материалов).
   - Есть seed-скрипт с тестовыми пользователями и демонстрационными данными для быстрого старта.

## Что не так / риски

1. **`next build` «успешен», но с runtime-ошибками Prisma при пререндере**
   - Во время сборки серверные страницы обращаются к таблицам, которых может не быть до `db push/seed`; в логах появляются ошибки `The table main.Person does not exist`, `main.Story`, `main.TimelineEntry`.
   - Это означает, что сборка не полностью «чистая» и поведение окружения сильно зависит от состояния локальной SQLite.

2. **Rate limit хранится в памяти процесса**
   - Лимитер реализован через `Map` в `globalThis`, что работает только в рамках одного процесса.
   - В serverless/multi-instance сценариях лимит будет обходиться, а после рестарта — сбрасываться.

3. **Поиск в архиве сделан в памяти, а не в БД**
   - Сначала загружаются все подходящие карточки, потом фильтрация по `q` идёт на Node.js.
   - На росте данных это даст лишнюю нагрузку на память/CPU и ухудшит latency.

4. **Нет централизованной CSRF-защиты для state-changing endpoints**
   - Есть `sameSite=lax`, но нет явного CSRF-токена/проверки origin для POST/PATCH API.
   - Для кабинета и административных маршрутов это желательно добавить.

5. **Загрузка файлов проверяет размер, но почти не ограничивает тип/содержимое**
   - `detectMediaType` ориентируется на `file.type`, который можно подделать на клиенте.
   - Файлы пишутся в `public/uploads/submissions`, то есть сразу становятся веб-доступными.

6. **Нет установленного ESLint-конфига (интерактивный `next lint`)**
   - Команда `npm run lint` запускает мастер настройки вместо линтинга, значит единый автоматический quality gate пока не настроен.

## Рекомендуемый приоритет исправлений

1. **P0:** гарантировать «чистую» сборку без Prisma-ошибок (инициализация БД до рендера, graceful fallback на пустые данные, health-check readiness).
2. **P1:** вынести rate limit в Redis/Upstash/DB с TTL.
3. **P1:** перевести поиск в `/archive` на SQL-фильтрацию (`contains`, индексы, пагинация).
4. **P1:** добавить CSRF-strategy (origin check + anti-CSRF token для мутаций).
5. **P2:** усилить upload security (allowlist расширений, проверка magic bytes, антивирусный pipeline, отдельный storage bucket).
6. **P2:** зафиксировать ESLint/TypeScript quality gate в CI.

## Итог

Для MVP проект выглядит структурно аккуратно: хорошая базовая схема, ролевая модель, модерация, аудит и валидация уже есть.

Главные слабые места — эксплуатационные: устойчивость сборки/рантайма при пустой БД, масштабируемость rate-limit/поиска и hardening безопасности для production-нагрузки.
